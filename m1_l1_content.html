1. Introduction

Welcome to Rust Programming on STM32. In this lesson, we embark on a journey that merges the reliability of modern systems programming with the ubiquitous ARM Cortex-M architecture. But before we write code, we must understand the "why."

What is Rust?

Rust is a multi-paradigm systems programming language designed for performance and safety, especially safe concurrency. Syntactically similar to C++, it guarantees memory safety by using a borrow checker to validate references at compile time, completely eliminating the need for a garbage collector.

History and Evolution

Rust was originally designed by Graydon Hoare at Mozilla Research in 2006. Mozilla officially sponsored the project in 2009, realizing its potential to rewrite rendering engines (Servo) safely. Rust 1.0 was released in May 2015. Since then, the Embedded Rust Working Group has aggressively built out the tooling required to target bare-metal architectures like the STM32. Today, the Rust Foundation (backed by AWS, Microsoft, Google, Huawei, and others) secures its future.

2. Why Embedded Systems Need Rust

Embedded systems control critical infrastructure: automotive brakes, pacemakers, and industrial robots. For decades, C and C++ have been the only viable choices due to their lack of runtime overhead. However, they place the entire burden of memory management on the developer.

The Problems with C/C++

  • Buffer Overflows: Writing past the bounds of an allocated array. In an STM32, writing past a buffer in SRAM might silently corrupt an adjacent RTOS task control block, leading to catastrophic failure hours later.
  • Null Pointers: Dereferencing a NULL pointer on a Cortex-M usually results in a HardFault.
  • Dangling Pointers: Accessing memory that has been freed.
  • Data Races: Two interrupts (or an interrupt and the main loop) accessing a shared global variable simultaneously without atomic guards or disabling interrupts.
// C Example: Buffer Overflow
void process_sensor_data() {
    uint8_t buffer[10];
    for(int i = 0; i <= 10; i++) { // BUG: Off-by-one error
        buffer[i] = read_adc(); // Corrupts adjacent memory on i=10
    }
}

In Rust, an array out-of-bounds access is caught at runtime (panics) or compile-time (if static), but it never silently corrupts memory.

3. Why C/C++ Is Still Dominant

If Rust is so safe, why does C still rule?

  • Legacy Code: Billions of lines of tested, certified C code exist.
  • Vendor SDKs: STMicroelectronics provides STM32Cube (HAL/LL), written entirely in C.
  • Toolchains: Established proprietary compilers (IAR, Keil) have decades of automotive certifications (ISO 26262).

However, the cost of debugging memory corruption in C is astronomical, prompting the industry's shift.

4. Why Rust Is Different

Rust solves memory issues at compile-time via three core rules of Ownership:

  1. Each value in Rust has a variable that’s called its owner.
  2. There can only be one owner at a time.
  3. When the owner goes out of scope, the value will be dropped.

Instead of passing ownership, you can Borrow data using references (&). You can have either one mutable reference (&mut T) OR any number of immutable references (&T). This completely prevents data races!

5. Memory Safety in STM32

An STM32 has Flash (for code and constants) and SRAM (for variables). In bare-metal Rust, we do not use a Heap. Everything is allocated on the Stack or statically.

Rust guarantees that if a function returns a reference to a variable allocated on its stack, the code will simply not compile. This prevents use-after-free errors natively.

6. Zero-Cost Abstractions

A zero-cost abstraction means that the high-level syntax you write compiles down to the exact same machine code you would have written in low-level C. You don't pay a runtime penalty for using higher-level constructs like Iterators or Enums.

Because Rust uses LLVM as its backend, it aggressively inlines functions and optimizes away unused branches.

7. Why the Automotive Industry Likes Rust

Automotive ECUs (Electronic Control Units), ADAS (Advanced Driver Assistance Systems), and V2X (Vehicle-to-Everything) communications require strict safety standards like ISO 26262. MISRA C exists to make C safer by banning unsafe features, but it is enforced via static analyzers. Rust enforces safety at the compiler level.

Companies like Volvo, Ford, and AUTOSAR consortium members are actively integrating Rust into Adaptive AUTOSAR to secure connected vehicles from cyber threats.

8. Rust on STM32 Ecosystem

The embedded Rust ecosystem consists of layers:

  • cortex-m: Core ARM architecture functions (NVIC, SysTick).
  • stm32f4 (PAC): Peripheral Access Crate, generated from ST's SVD files using svd2rust. Gives safe register access.
  • stm32f4xx-hal (HAL): High-level API for GPIO, I2C, SPI.

10. Knowledge Check

1. What does the Borrow Checker prevent?
A) Compilation
B) Data Races and Memory Corruption
C) Stack Allocation
Answer: B

2. True or False: Rust requires a Garbage Collector for embedded systems.
Answer: False. Rust uses Ownership to manage memory without a GC.

3. Interview Scenario: "Why shouldn't we just use MISRA C instead of learning Rust?"
Answer: MISRA C relies on external static analysis and developer discipline. Rust enforces memory safety natively at the compiler level, preventing unsafe code from ever compiling.

In this lesson, we established that C/C++'s lack of memory safety is a massive liability in modern, connected embedded systems.
